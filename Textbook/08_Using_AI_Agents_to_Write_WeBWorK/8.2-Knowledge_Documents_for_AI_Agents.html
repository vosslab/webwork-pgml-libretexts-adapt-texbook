Without WeBWorK-specific documentation, AI agents fall back on their general Perl training data and write code that looks syntactically correct but fails inside the PG sandbox. Curated reference documents prevent this. The analogy is straightforward: you would not hand a lab technician a set of reagents without a protocol and expect a reliable result. The same principle applies to AI agents. As one practitioner put it: "all the documents it needs is in the skill. So please don't search my hard drive."

This page explains what knowledge documents are, introduces a ready-to-install skill that contains them, describes what those documents cover, and shows how to build your own set if you prefer to customize.

The WeBWorK Writer skill (ready to install)

The fastest path for most users is to install a pre-built skill that already contains curated WeBWorK authoring documents. The webwork-writer skill is publicly available and maintained alongside this textbook.

Claude Code install (one command): 

claude plugin add https://github.com/vosslab/vosslab-skills/vosslab-skills:webwork-writer

GitHub repository: The source documents are hosted at https://github.com/vosslab/vosslab-skills/tree/main/skills/webwork-writer

For Codex users: Download the reference documents from the same GitHub repository and add them to a forum in the Codex interface.

The skill contains approximately ten reference documents covering PGML structure, common mistakes, macro allowlists, renderer API usage, CSS-based color and emphasis, advanced randomization, matching problems, niceTables, linting rules, and OPL header metadata. Installing this single skill gives the agent everything it needs to write most WeBWorK problems correctly, with no manual document assembly required.

Why agents need WeBWorK-specific documents

PG looks like Perl, but it is not standard Perl. Agents trained on general Perl code will confidently use patterns that break inside the PG sandbox. Here are the most common traps:

  • PG runs in a sandbox that blocks most Perl builtins. Calls to open, system, and die will fail silently or throw errors.
  • The my keyword breaks radio buttons and checkboxes. PG scoping rules differ from standard Perl, and lexical variables declared with my become invisible to the answer evaluator.
  • Backslash references like \@array behave differently inside PGML than in regular Perl.
  • PGML is single-pass. You cannot nest PGML commands or use Perl control-flow statements (like if or for) inside a BEGIN_PGML block.
  • Standard HTML tags are blocked in PGML. Use CSS classes injected via HEADER_TEXT instead.
  • Colors must use CSS, not TeX \color{} commands. The TeX color commands are escaped before MathJax processes them.
  • Only PG 2.17 macros are available. Newer Perl modules are not loaded in the renderer.

Without documentation that explains these differences, agents will hit every one of these traps. For detailed examples and fixes, see Common Mistakes and How to Fix Them and Common PG Macros.

Building your own document set

If you want to customize the reference material or use an agent that does not support skills, you can assemble your own document set. Keep these guidelines in mind:

  • Keep all documents in a single directory so the agent does not search your entire filesystem. A dedicated folder like webwork-docs/ works well.
  • Include an explicit instruction in your agent configuration: "Use only the provided documents. Do not search for other files."
  • Start with a minimal set and add documents only as you encounter gaps. Four documents are enough to get started: a PG authoring guide, the common mistakes list, the macro allowlist, and the renderer API reference.
  • When you find a new trap or pattern, add it to your documents instead of relying on the agent to remember it from a previous conversation.

The textbook pages listed in the table above are a good source for building your own documents. You can adapt their content into plain-text or Markdown files that your agent can read.

Apply it today

  • Install the webwork-writer skill with one command: 
    claude plugin add https://github.com/vosslab/vosslab-skills/vosslab-skills:webwork-writer
  • Test with a simple problem. Ask your agent to write a multiple-choice question about your favorite color. If the agent produces a working PG file that renders without errors, the skill is installed correctly and the agent can find the documentation.
  • Once the skill is confirmed, move on to Connecting AI Agents to the PG Renderer so the agent can compile and test its own output automatically.
