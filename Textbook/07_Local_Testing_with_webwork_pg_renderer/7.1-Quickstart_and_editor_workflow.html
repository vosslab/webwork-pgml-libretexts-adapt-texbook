Local rendering works best when it feels like editing a document: open a problem file, render it, make a small change, and render again. This section outlines a simple workflow you can repeat every day.

Choose one file and make it reproducible

  • Pick one .pg file to work on and keep the path stable while you iterate.
  • Use a fixed seed when you are debugging so the same random variant renders each time.
  • Change one thing per iteration so you can tell what caused the difference.

Render and check the student-facing prompt

  • Render the problem and read it like a student would read it.
  • Confirm the story makes sense with the chosen randomized values (units, ranges, and rounding).
  • Confirm every answer blank is visible, in the right order, and has the right unit/format cues nearby.

Save edits back to disk

  • Keep edits small and frequent rather than making one large rewrite.
  • When you find a failing case, keep the file path and seed so you can reproduce it later.
  • Once the local render looks right, move to course-level testing for the final check.

Apply it today

  • Use local rendering for the tight edit-preview loop, then use ADAPT for the final publish workflow.
  • When debugging, write down the failing seed and do not change it until the failure is fixed.
